AI/LLM
Layer
Intent Classification
A fast Mistral 7B call classifies the question. Is it a DATA query (needs SQL pipeline) or CHAT (direct conversational reply)? This single routing decision changes the entire execution path.
app/llm/intent_classifier.py app/llm/gemini_client.py
Function: classify_intent(question) → "data" | "chat"
→ Classifier Prompt
Classify: "Show unpaid invoices above 5 lakhs from last quarter" Reply with one word: DATA or CHAT
← Classification Result
Intent: DATA → Routes to full SQL pipeline (CHAT → direct Mistral reply, no SQL, no DB execution)
↓ run_query() starts SQL pipeline
3
⚡
Cache
Layer
Semantic Cache Lookup
Before calling the LLM for SQL, ChromaDB checks if a semantically similar question was asked before. If cosine similarity ≥ 0.92, return verified cached SQL instantly — no LLM call needed.
app/services/cache_service.py
Function: get_cached_sql(question) → sql | None
→ ChromaDB Query
embedding("Show unpaid invoices...") ↓ cosine similarity search threshold: 0.92 checking 10 cached queries...
← CACHE HIT ⚡
similarity: 0.97 ✓ Verified SQL retrieved Steps 4–8 SKIPPED → Jump directly to execution
Cache MISS path: Steps 4–8 run (schema retrieval → graph → prompt → LLM → validate). Cache HIT skips all of that.
↓ (cache miss path) schema retrieval

Schema Retrieval (Cache Miss Only)
sentence-transformers embeds the question into a vector. ChromaDB finds the top-5 most semantically relevant tables from 15 (scales to 800+). Only these schemas are sent to the LLM — preventing hallucination.
app/schema/schema_manager.py database/enterprise_schema.py
Function: get_relevant_tables(question, top_k=5)
invoices
relevance: 0.94
customers
relevance: 0.91
payments

sales_orders
relevance: 0.82
contracts
relevance: 0.74
Why this matters: Sending only 5 relevant table schemas (not all 800+) prevents the LLM from hallucinating columns/tables that don't exist.
↓ discover_joins(table_names)

Graph
Reasoning
Graph-Based Schema Reasoning
NetworkX builds a directed graph from all foreign key relationships. When relevant tables are found, the shortest JOIN path between them is computed automatically — no hardcoded JOINs anywhere.
app/schema/schema_graph.py
Function: discover_joins(table_names) → list[str]
→ FK Graph (NetworkX DiGraph)
invoices ──FK──→ customers invoices ──FK──→ invoice_items sales_orders ──FK──→ invoices payments ──FK──→ invoices 15 tables, 15 relationships
← Discovered JOIN conditions
invoices.customer_id = customers.customer_id Shortest path length: 1 Injected into prompt ✓
Bonus feature: Graph-based schema reasoning + autonomous join discovery. No hardcoded JOIN logic anywhere in the codebase.
↓ build_sql_prompt() with join_hints

Adaptive Prompt Building
The prompt is dynamically assembled — not static. It includes the retrieved table schemas, auto-discovered JOINs from the graph, business glossary, and multi-turn conversation history. Every query gets a custom prompt.

7
🤖
Mistral
7B LLM
SQL Generation — Mistral 7B
Ollama runs Mistral 7B locally. The adaptive prompt is sent. If the generated SQL fails validation, an automatic correction prompt is sent (up to 3 retries). Confidence score decreases with each retry.
app/llm/gemini_client.py Ollama → mistral:7b
Function: generate_with_retry(prompt, max_retries=3, temperature=0.1)
→ Retry Mechanism
Attempt 1 → generate SQL → validate_sql() → ✓ pass Confidence: 1.0 (if fail → correction_prompt Attempt 2 → confidence: 0.85 Attempt 3 → confidence: 0.70)
← Generated SQL (raw)
SELECT i.invoice_id, c.customer_name, i.total_amount, i.due_date, i.status FROM invoices AS i JOIN customers AS c ON i.customer_id=c.customer_id WHERE i.total_amount > 500000 AND i.paid_amount < i.total_amount ORDER BY i.total_amount DESC LIMIT 100
↓ validate_sql(sql)
8
🛡
Safety
Checks
SQL Validation & Safety Layer
Every generated SQL passes through a strict safety validator before touching the database. Blocks all destructive operations. Strips markdown wrappers. Auto-adds LIMIT if missing. Parses with sqlparse to verify structure.
app/validation/sql_validator.py
Functions: extract_sql_from_response(raw) → validate_sql(sql)
  • ✅ Starts with SELECT (not DROP/INSERT/UPDATE)
  • ✅ No DROP, DELETE, TRUNCATE, ALTER keywords
  • ✅ No UNION SELECT injection patterns
  • ✅ LIMIT 100 auto-added if missing
  • ✅ sqlparse confirms valid query structure
↓ execute_query(validated_sql)

Database
Layer
Database Execution (SQLAlchemy)
SQLAlchemy executes the validated SQL on the SQLite enterprise database. 15 tables, 830+ rows of realistic Indian enterprise data across Finance, Sales, HR, Procurement, and Inventory domains.
